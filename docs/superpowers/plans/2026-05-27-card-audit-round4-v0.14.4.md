## Per-task workflow shape

Every Phase-1-through-4 task follows this 7-step pattern. Steps are spelled out per task below, but the shape is constant — refer back here if a task's intent isn't obvious:

1. **Re-audit the trigger card after Phase 0.** Run `node /Users/Dada/.claude/skills/mtg-graph-card-tag-audit/card_lookup.mjs "<Card Name>" --normalized` to confirm the issue still exists in the fresh artifact. If the tag now fires, the task is obsolete — strike it from this plan and move on. (Phase 0 cleared 12+ stale-artifact entries; verify, don't trust.)
2. **Add failing fixture rows** to the rule's `.test.ts` `it.each(...positive)` block. Use the normalized text from step 1 verbatim (the lookup output's "Normalized text" section).
3. **Run the tests:** `npm test -- <rule-id>.test.ts`. Expected: the new rows fail.
4. **Broaden the regex** in `<rule>.ts` per the task's "Regex change" section. Keep the change minimal — one new alternation or one new lookbehind, never a rewrite.
5. **Re-run the tests:** `npm test -- <rule-id>.test.ts`. Expected: all rows pass, no negative-case row regresses.
6. **Run coverage to spot-check the broadening:** `npm run rule:coverage -- <tagId>`. Expected: match count went up by the expected order of magnitude (1 card for a niche fix, 10s for a family fix); inspect 10 random positives for false positives the regex shouldn't have admitted.
7. **Commit** with message `fix(<rule-id>): <one-line summary>` per the project's commit-style convention (see `git log --oneline pipeline/rules/`).


- [ ] **Step 4: Broaden `SELF_ETB` regex**

Edit `pipeline/rules/trigger.self_etb.ts:19-24`. Replace:

```typescript
const SELF_ETB = new RegExp(
  '\\b(?:when|whenever) ' +
  '(?:__self__|this (?:creature|artifact|enchantment|land|permanent|planeswalker|vehicle|saga|aura|room|equipment|battle|case|class))' +
  ' (?:enters(?:\\s+or attacks)?(?!\\s+with)|attacks or enters)' +
  '\\b[^.]*?,',
);
```

With:

```typescript
const SELF_ETB = new RegExp(
  '\\b(?:when|whenever|as) ' +
  '(?:__self__|this (?:creature|artifact|enchantment|land|permanent|planeswalker|vehicle|saga|aura|room|equipment|battle|case|class))' +
  ' (?:enters(?:\\s+or (?:attacks|is turned face up))?(?!\\s+(?:with|tapped|untapped)?\\s*[,.])|attacks or enters)' +
  '\\b[^.]*?,',
);
```

The two changes:
1. `(?:when|whenever)` → `(?:when|whenever|as)` (item B verb broadening)
2. `enters\\s+or attacks` → `enters\\s+or (?:attacks|is turned face up)` (Disguise face-up combined trigger)
3. Lookahead `(?!\\s+with)` extended to `(?!\\s+(?:with|tapped|untapped)?\\s*[,.])` — blocks "as X enters tapped," and "as X enters untapped," replacement effects. Comma/period anchor required since replacement effects always end the clause with a punctuation mark before the modifier.

- [ ] **Step 3: Run, verify failures**

- [ ] **Step 4: Add new frame to `buildGrantRegex`**

In `effect.grants_keyword.ts`, the `buildGrantRegex(kw)` function builds a regex from ~13 frames (lines 44–113). Add a 14th frame after Frame (f2) at line 89, modeled on the existing list-style frames:

```typescript
      // Frame (j): "gains your choice of <kw1>, <kw2>(, ... )?, or <kw>" — multi-
      // keyword choice grant. Ezrim Agency Chief: "gains your choice of vigilance,
      // lifelink, or hexproof". Each listed keyword fires independently. The kw
      // can be at any list position; we match if it's anywhere in a comma-list
      // bounded by "your choice of" on the left and "until end of turn" or a
      // period on the right.
      `\\bgains?\\s+your choice of\\s+(?:[a-z]+(?:\\s+strike)?\\s*,\\s+){1,4}(?:and\\s+|or\\s+)?${kw}\\b`,
      // Frame (j2): mid-list — the kw is NOT the final item. "...your choice of
      // vigilance, lifelink, or hexproof" — when kw=vigilance, kw is the first item.
      `\\bgains?\\s+your choice of\\s+(?:[a-z]+(?:\\s+strike)?\\s*,\\s+)*${kw}\\s*,`,
      // Frame (j3): 2-item choice, no comma. "gains your choice of trample or lifelink".
      `\\bgains?\\s+your choice of\\s+(?:${kw}\\s+or\\s+[a-z]+|[a-z]+\\s+or\\s+${kw})\\b`,
```

The three sub-frames cover the three positions a keyword can occupy in a choice list (terminal, mid-list, or 2-item-no-comma). This is intentionally redundant — each frame fires independently and the regex `|` short-circuits at the first match.
